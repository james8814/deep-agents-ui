# 修复方案

## 问题分析

经过完整代码审查，确认其他团队已修复了以下问题：

- `client.ts` 已添加 `getStoredToken()` 自动注入 Bearer Token（第 79 行，第 85 行）
- `fetchInterceptor.ts` 已扩展覆盖 Auth Server `/api/` 端点（第 39-47 行）

### 仍存在的问题

**问题 1：`/auth/me` 401 控制台报错**

原因：用户有过期 Token 时，`AuthContext.checkAuth()` 调用 `getUserInfo(storedToken)` →
`fetchWithCredentials` 发送请求 → 服务端返回 401。

虽然 AuthContext 的 catch 块正确处理了这个错误（清除 token、跳转登录页），但浏览器控制台仍然显示红色的 `Failed to load resource: 401` 错误。这是浏览器原生行为，无法抑制。

但 `fetchWithCredentials` 的 401 错误处理路径有问题：

- 第 96-98 行：`!url.includes("/auth/")` 条件会跳过 `/auth/me` 的 token 刷新
- 这意味着 `/auth/me` 的 401 直接抛出 `Error(error.detail)` (第 126 行)
- 但 error.detail 是后端返回的字符串如 "未登录或 Token 已过期"
- AuthContext 靠 `error.message.includes('未登录')` 来判断是否是认证错误（第 77 行）

**风险**：如果后端返回的 detail 文案变化，AuthContext 的字符串匹配可能失败，导致 token 不被清除。

**修复**：让 `fetchWithCredentials` 在抛出 `/auth/` 端点的 401 错误时，包含状态码信息。

**问题 2：`BlockingError: An internal error occurred at StreamManager.enqueue`**

原因：`useChat` hook 中的 `useStream` 在组件挂载时会立即尝试 `reconnectOnMount: true`（第 52 行）。
当 token 还没准备好（刚从过期 token 清除后重新登录），或者 `activeAssistant` 为 null 时，
`useStream` 发送的请求可能失败，触发 SDK 内部的 StreamManager 错误。

虽然 AuthGuard 会阻止未认证用户渲染页面内容，但问题在于：

- `HomePageContent` 在 AuthGuard 内部渲染
- `ClientProvider` 使用 `tokenRef` 模式，初始值可能为 null
- `useStream` 的 `assistantId` 为空字符串时（assistant 还没加载），SDK 可能尝试无效请求

**修复**：在 `useStream` 传入空 `assistantId` 时禁用自动连接。

## 修复清单

### Fix 1: `src/api/client.ts` - 改善 401 错误信息

在 `/auth/` 端点的 401 错误中包含 HTTP 状态码，确保 AuthContext 可靠地识别认证错误。

### Fix 2: `src/contexts/AuthContext.tsx` - 用 HTTP 状态码判断认证错误

不再依赖脆弱的字符串匹配，改为检查标准化的错误格式。

### Fix 3: `src/app/hooks/useChat.ts` - 禁用空 assistant 时的自动连接

当 `assistantId` 为空或 client 未准备好时，不启用 `useStream`。
